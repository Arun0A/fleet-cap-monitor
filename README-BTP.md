
# Fleet CAP project — SAP BTP deployment notes

This file contains quick instructions to deploy the Fleet CAP app to SAP BTP (Cloud Foundry).

Prerequisites
- An SAP BTP account (trial or paid) with Cloud Foundry enabled
- `cf` CLI installed and logged in (`cf login -a <API> -u <user> -o <org> -s <space>`)
- `cf` service instances created for HANA (or HANA Cloud), XSUAA and Event Mesh (or use equivalent names)
- Node.js toolchain for local testing (prefer inside `nix develop` created for this repo)

Quick local dev
1. Enter the dev shell from repository root and start the app + worker:
```zsh
nix develop
cd o2c-cap-project
npm install
# Start cap app (in dev mode)
npm run watch

# In another terminal start the worker that auto-creates work orders from alerts
cd worker/invoice-worker
npm install
npm start
```

Build & deploy to Cloud Foundry
1. Build the CAP project for deployment:
```zsh
cd o2c-cap-project
npm run build   # runs `cds build` -> produces `gen/` directory
```
2. Create or bind required service instances in your CF space (`fleet-hana`, `fleet-xsuaa`, `fleet-eventmesh` are example names). Use the BTP cockpit or `cf create-service`.
3. Push the app + worker using the provided `manifest.yml`:
```zsh
cf push -f manifest.yml
```

Notes
- `xs-security.json` contains example roles and scopes. Create an XSUAA instance and bind it to the app; adapt `xs-security.json` if you want different role names.
- `manifest.yml` references `gen/srv` as the app path generated by `cds build`, and `worker/invoice-worker` for the background worker. Adjust `path` values if needed.
- Event Mesh is optional for the MVP; if not used the app still runs with in-process event handling and the worker can poll Alerts.

Events / Event Mesh
- The project now emits an `AlertCreated` event when telemetry crosses thresholds. The behavior is:
  - In-process: `srv.emit('AlertCreated', payload)` is emitted and handled by an in-process subscriber that auto-creates a WorkOrder.
  - Worker polling: a background worker polls `Alerts` and creates `WorkOrders` for any unhandled alerts.
  - External (BTP/Event Mesh): when deployed and bound to Event Mesh the app attempts to publish the `AlertCreated` event.

Testing events locally
- By default the dev server allows in-process events. To simulate production enforcement, set `CDS_ENV=cf` when starting the app.
- To test external messaging in a BTP deployment, bind an Event Mesh service instance and ensure the `messaging` connection name is available in `vcap_services` (or service bindings).
